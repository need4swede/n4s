Frequently Asked Questions

Does it work on M1 Macs?

Yes — podRacer has native support for both x86/x64 (Intel) and arm64 (M1) instruction sets.
You don't need to use Rosetta to run it.

Is there a Windows version?

Currently, no — but a Windows 10/11 version for both x86 and arm64 is in development.
No estimated release date for now.

Will it work on older versions of MacOS?

I have tested it on Mojave 10.14.6+ and can confirm that it works without any issues. There really isn't much in the way of ensuring that it'll work on older operating systems, but there is one caveat - expired certifications. The DST Root CA X3 certification expired on September 30, 2021. That means those older devices that don’t trust ISRG Root X1 will start getting certificate warnings when visiting sites that use Let’s Encrypt certificates. This could cause some parts of the application to not function properly.

If you're effected by this, you can update your certification by downloading a new one:

    1. Make a copy of your existing cert

    
sudo cp /etc/ssl/cert.pem{,-orig}

    2. Download the new certificate (04/26/22)

    3. Rename the new cert to: cert.pem

    4. Replace your old cert with the new one

    
sudo mv cert.pem /etc/ssl/cert.pem

Can this parse paid-only content?

Only if it has access to the XML data of the paid content. podRacer isn't partial to one form of podcast or another – it will handle them all the same. So if you have an RSS feed with premium content, even one that uses token authorization, podRacer will be able to retrieve the content just fine.

Will it work with Patreon?

Yes — given that you're able to provide podRacer with a valid RSS feed, it can handle token authorization and retrieve content from Patreon accounts just fine. The RSS feed needs to include both the token-hash and the token-time properties for this to work.

What happens if I already have a podcast downloaded?

Unless you explicitly enable the overwrite setting, podRacer doesn't delete/overwrite any existing media. What it does do is check to make sure that the media is valid by reading the properties of each file first and deciding if whether or not it needs to be updated. If you have an .mp3 file called 'audio.mp3', and you end up editing its content, podRacer will update the file upon running a new download task. To prevent this, you can move the file out of the podcast's media directory and it won't be affected

Otherwise, if podcast content already exists and it matches the content from the podcast host, podRacer will automatically skip downloading that file and will move on to the next one. Enabling the overwrite setting will replace every file affected in the download task.

Can I add a provider to 'Find Podcast'?

Apple Podcast is the only supported podcast provider at the moment. This may change in the future, but it's not a major area of concern. You can always use the metadata reports to search for the content using your prefered provider at any time.

Reported Bugs

Unable to run installer Unresolved Solvable

MacOS, by default, prevents you from running the application because it can't verify that it came from a trustworthy source, i.e. the App Store. You can bypass this by right-clicking on the installer, and clicking 'Open' — or by allowing the app to run via System Preferences > Security & Privacy > 'Open Anyway'

Strange characters in metadata Resolved in v1.2

This is an encoding issue where certain special characters are not being properly 'translated' to their ASCII counterparts.
I will address this in the next release.

App is unresponsive during downloads Resolved in v1.0

This is happening due to the way the process is actually running. It is currently running within the UI object, so the UI is actually on-hold until the request returns a value. This causes the UI to 'hang' in between the download requests, but the application is still very much alive and running.

I am currenly re-writing the download process to be multi-threaded. By using a threadpool, the requests are handled by a seperate process in the background, emitting signals back to the parent object instead of running in the UI itself. This will not only resolve the UI 'freeze' bug, but it will also speed up functionality along the way.
